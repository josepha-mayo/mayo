## YOUR TASK
Perform a deep, multi-layered analysis and produce a structured improvement plan.

### Step 1: Codebase Understanding
- What does this repository do? (1-2 sentences)
- What does the target file do within the repo?
- What patterns, frameworks, or conventions does the codebase use?

### Step 2: Deep Analysis
Scan for ALL of the following (big AND small):
- **Security**: Injections, hardcoded secrets, missing input validation, auth bypasses
- **Logic**: Off-by-one errors, missing null checks, edge cases, incorrect return types, race conditions
- **Performance**: Unnecessary loops, redundant API calls, memory leaks, N+1 queries
- **Architecture**: Missing error handling patterns, inconsistent API responses, tight coupling
- **Features**: Missing functionality that would clearly improve the project
- **Testing**: Missing validation, unchecked user input, error paths without handlers
- **DX (Developer Experience)**: Missing build/run instructions, unclear docs
- **Consistency**: Naming conventions, import patterns, style mismatches
- **Dead Code**: Unreachable branches, unused variables, obsolete functions

### Step 3: Pick ONE Improvement
CRITICAL: You are a Senior Engineer. Only propose changes that meet ALL of these criteria:
1. **MUST be a real bug or meaningful improvement** — not style, not docs, not trivia
2. **MUST fix actual broken functionality** — not just "could be better"
3. **MUST have measurable impact** — security fix, crash prevention, real performance gain

REJECT trivial changes:
- ❌ Typo fixes (unless it breaks the code)
- ❌ Comment/doc changes
- ❌ Whitespace/formatting
- ❌ Variable renaming
- ❌ "Better naming" suggestions
- ❌ README updates
- ❌ Package version bumps (forbidden anyway)
- ❌ Adding TODOs or comments
- ❌ Extracting constants that aren't actually used differently

ONLY proceed if the fix is:
- ✅ Fixes a runtime error or crash
- ✅ Closes a security vulnerability
- ✅ Fixes broken/missing error handling
- ✅ Removes genuinely dead code
- ✅ Fixes a logic bug
- ✅ Fixes N+1 or memory leak
- ✅ Fixes missing input validation

### Step 4: Write the Executor's Plan
CRITICAL: Your plan must be SPECIFIC enough for the Executor to find EXACT code:

- **WHAT** to change (describe in words, do NOT write code)
- **WHERE** in the file(s) — give EXACT function name, variable name, or code snippet they can search for
- **WHY** this matters (technical justification)
- **HOW** to do it (step-by-step logic, NOT code)

IMPORTANT: If the code location is ambiguous (e.g., "fix the API handler"), your plan is TOO VAGUE. The Executor will fail. Be specific:
- ✅ "In the `fetchUserRepos` function in `apps/web/src/app/profile/[username]/repos/page.tsx`, around line 45 where it calls `repo.issues`"
- ❌ "Fix the user repos page"
